# VLM 提示词自动优化技术调研

> 本文定义 Prompt Loop 从“支持多模态样本的文本改写”走向“VLM 解决方案自动优化”的技术路线。产品范围见 [PRD](./prd-smart-prompt-optimize.md)，当前实现见 [TRD](./trd-smart-prompt-optimize.md)。

## 1. 第一性原理与能力边界

闭源 VLM 的视觉编码器通常不可修改，Prompt 无法直接改变模型已经编码出的视觉 token。但系统仍可通过两类手段改变结果：

1. **视觉输入变换**：裁剪、放大、分块、对比度增强、OCR、检测框或 Set-of-Mark，让关键证据以更高有效分辨率和更低视觉噪声进入模型。
2. **跨模态提示**：空间扫描、先感知后推理、引用区域编号、图文 Few-shot，让语言模型更稳定地检索和利用已经进入上下文的视觉证据。

因此，自动优化对象不应只是一个字符串，而应逐步扩展为：

```text
SolutionPlan
  ├─ TextPromptPlan       指令、角色、输出协议、推理阶段
  ├─ VisualTransformPlan  裁剪、缩放、分块、标记及参数
  ├─ ExamplePlan          图文示例选择、顺序和对比示例对
  └─ PipelinePlan         单阶段 / 定位后裁剪 / 多视图聚合
```

每个方案必须可版本化、可复现、可回滚，并在同一冻结验证集上与原方案比较。

## 2. 多模态错误诊断探针集

探针的目标不是直接生成新 Prompt，而是先回答“错误发生在输入、感知、推理还是输出协议”。

| 探针 | 执行方式 | 关键信号 | 分流动作 |
|------|----------|----------|----------|
| 输入完整性 | 检查 runtime message parts、媒体类型、字节数、尺寸和可解码性 | 图片未进入模型、顺序错、过度压缩 | 修数据与适配器，不改 Prompt |
| 裸描述 | 不带任务结论要求，仅让同一 VLM描述关键区域和属性 | 关键目标完全未被提及 | 进入视觉变换或空间引导通道 |
| 定位 | 要求输出目标框、区域编号或方位 | 找到目标但位置漂移 | 分块、放大、SoM、坐标引导 |
| 感知—推理解耦 | 第一步只描述证据，第二步仅据描述判断 | 描述正确但判断错误 | 改任务定义、判别规则或 Few-shot |
| 输出协议 | JSON/Schema、枚举、必填字段和拒答短语硬校验 | 语义似对但协议破坏 | 结构化约束、重试或直接记 0 分 |
| 对照扰动 | 遮挡非关键区、替换文本问题、交换相似样本 | 答案随语言先验而非证据变化 | 加强证据引用和反事实约束 |
| 跨模型复核 | 目标 VLM 与独立 Judge/规则交叉检查 | 单裁判全满分或批次漂移 | 触发复核、拆批或人工检查 |

### 2.1 落地数据结构

```text
ProbeResult
  case_id
  probe_type
  status: pass | fail | uncertain
  evidence[]              # 区域、描述、硬校验错误，不保存密钥
  failure_stage: input | perception | reasoning | protocol
  confidence
  recommended_actions[]
  model_and_media_snapshot
```

Worker 先运行低成本确定性探针，再按需调用 VLM 探针。探针结果写入任务报告，用于生成候选和解释“为什么改”。同一 case 的媒体只解码一次，探针应支持批量执行与缓存。

## 3. VLM 解决方案自动优化闭环

```text
冻结 Prompt、模型、样本和评判规则
  → 执行原方案 baseline
  → 对失败样本运行诊断探针
  → 按 failure_stage 生成候选 SolutionPlan
  → 在优化集执行候选
  → 同一 Judge + 硬规则评分
  → 在冻结验证集比较 candidate - baseline
  → 满足质量、成本、延迟护栏才推荐，否则拒绝
```

候选生成采用有边界的搜索空间：

- `input` 失败：只修媒体获取、编码和消息组装。
- `perception` 失败：空间扫描、局部裁剪、分块、多视图或 SoM。
- `reasoning` 失败：感知—推理解耦、判别规则、反例和对比 Few-shot。
- `protocol` 失败：Schema、枚举约束、结构化重试和确定性解析器。

禁止在一次候选中无记录地同时改变模型、Prompt、图像和 Judge，否则无法归因。联合优化时必须把每个变换及参数写入 `SolutionPlan`。

## 4. VLM 特有策略

### 4.1 感知—推理解耦

先要求模型描述可观察证据，再基于已生成描述判断。它把“视觉 token 直接到结论”改为“视觉证据显式化后再推理”，适合处理语言先验压过视觉信息的问题。

### 4.2 空间引导与 Set-of-Mark

空间引导要求按左到右、上到下或指定方位扫描。SoM 则在派生图上添加稳定编号，并在文本中逐一引用。后者必须作为“图像派生物 + Prompt 引用”的原子变更保存，不能只记录改写后的文字。

### 4.3 图文 Few-shot

VLM Few-shot 不只教输出格式，还教“如何看”。示例选择需考虑视觉同构、答案中的视觉理由和正负对比对。由于视觉 token 成本高，默认策略应少而准，并在验证集评估示例污染和过拟合。

### 4.4 Coarse-to-Fine

第一阶段定位疑似区域，第二阶段从原图裁剪高分辨率局部再识别。它用额外调用换有效像素密度，适用于小目标缺陷；需要坐标合法性、边界扩张、派生图缓存和失败回退。

## 5. 裁判与选优保护

LLM Judge 不是事实源，至少需要以下保护：

- 原方案与候选使用同一裁判模型、温度、规则、参考答案和样本快照。
- JSON/Schema、枚举、必填项、拒答和空输出先做确定性校验；硬失败记 0 分或重试。
- 对全满分、分数无方差、批次间突变等可疑结果拆批复核。
- 排名使用验证集相对增益，不使用候选绝对分；同时设置成本与延迟护栏。
- 自动推荐不等于自动发布，结果先回填草稿并保留人工确认。

## 6. 与当前实现的差距和建议顺序

| 阶段 | 状态 | 交付 |
|------|------|------|
| M0 多模态执行闭环 | 已完成 | EvaluationSet Items、multipart 保真、原 Prompt baseline、批量 Judge、验证集相对增益 |
| P0 裁判一致性 | 已完成并部署回归 | 参考 JSON 结构硬校验、拒答检测、可疑全满分复核、硬失败归零；无增益终态 `no_gain` |
| P1 诊断探针 | 待做（研究侧下一优先；产品侧当前高优见 HANDOFF P1 UI） | 输入完整性、裸描述、定位、协议探针与 `ProbeResult` |
| P2 文本策略路由 | 待做 | 根据感知/推理/协议错误定向生成候选，而非统一改写 |
| P3 视觉变换 | 待做 | 可复现裁剪/放大/分块/SoM、派生媒体存储和审计 |
| P4 联合方案搜索 | 待做 | `SolutionPlan` 搜索、质量/成本/延迟多目标选优 |

P0 已完成代码级保护，并完成无增益 `no_gain` 部署回归。可选再用已知协议变形样本做一次真图硬校验回归。产品侧下一优先是 HANDOFF 的 P1 UI；研究侧再进入诊断探针。仍应持续观察 Judge 与硬规则冲突率以及全满分复核命中率。

## 7. 验收指标

- 感知失败、推理失败、协议失败的诊断准确率与不确定率。
- 验证集相对基线增益，以及无提升候选的拒绝率。
- Judge 与硬规则冲突率、可疑满分复核命中率。
- 单任务模型调用数、视觉 token、耗时和资源点。
- 视觉变换方案的可复现率、回滚成功率和派生媒体缓存命中率。
- 人工采纳率与采纳后线上回归率。

## 8. 技术版图定位

文本 APO、ProTeGi/文本梯度、OPRO、DSPy/MIPRO 可用于“怎么想、怎么写”的候选搜索；VLM 方案还必须覆盖“怎么看”：输入完整性探针、视觉注意引导、图文示例检索、可复现视觉变换和多阶段管线。Prompt Loop 的差异化方向不是简单移植 LLM Prompt 改写，而是用诊断探针把 badcase 分流到文本侧或视觉侧，并在统一评测闭环中验证整个 `SolutionPlan`。
